Two doors into the same data: MCP vs. REST API

A hands-on comparison of the two ways to pull AI brand-visibility metrics out of Rankscale — a purpose-built MCP server and the raw REST API — measured on capability, correctness, and the metric that quietly decides cost at scale: token usage.

Subject: Rankscale Brand under test: De'Longhi (US) Method: paired sub-agents, identical model Updated 2026-07-13
MCP — the connector REST API — the skill

Key findings

The short version

Both surfaces read the same underlying data and produce the same answers. The real decision is about shape of work and token cost — and there the two swap places depending on how much data you touch.

The headline

Token efficiency crosses over with payload size

For a single narrow metric, the MCP is markedly leaner (~40k vs ~56k tokens). For a broad pull across many topics and competitors, the REST API wins by roughly 2× (~65k vs ~128k tokens). There is no universal winner — there is a break-even point, and it sits at how much data the task drags into context.

Correctness

Same data, same answers

Where scopes match, both surfaces return identical visibility values and identical brand ranks. Divergences appeared only in the deep long tail, from competitor-pool size — not from either tool being wrong.

Capability

The asymmetry is writes vs. reach

Only the API can write (manage terms, spend credits, run prompts), read raw AI answer text, and hit arbitrary date windows. Only the MCP exposes shopping, per-term sources, and saved dashboard views as first-class reads.

Gotcha

“Brand rank” is never handed to you

Neither surface returns a ready rank. The MCP even has an own_brand_rank field that is always 1 — a display artifact, not a ranking. Rank must be computed from visibility, or you will silently report a brand as #1 everywhere.

Identity

The two doors can open different rooms

The MCP connector and the REST key authenticate independently. In testing they resolved to different workspaces — a key that works for one brand can be blind to another. Confirm which account each surface is bound to before trusting a blank result.

Background

Two ways to reach the same numbers

Rankscale tracks how brands surface inside AI assistants — it feeds tracked prompts to ChatGPT, Gemini, Claude, Perplexity and others, then scores how often, where, and how favourably each brand is mentioned. Analysts pull that data out for reporting in one of two ways.

The MCP connector

A set of read-only tools an AI agent can call directly. Responses are pre-digested, capped, and safety-wrapped. Tool schemas load on demand, so the ramp-up cost is tiny — but every response lands straight in the model's context.

The REST API (via a skill)

Raw HTTP endpoints called with curl. Full read and write. Responses are large and monolithic, but the agent writes them to disk and extracts only what it needs — keeping bulk payloads out of context at the cost of a heavier instruction set to learn first.

For an SEO reader, the practical translation: the MCP is the quick-question tool that an assistant reaches for mid-conversation; the API is the reporting pipe you build a dashboard or a bulk export on. The rest of this document shows exactly where each pays off.

Detailed breakdown · 02

The token model — why the winner flips

Token usage is governed by two costs pulling in opposite directions: a fixed ramp cost to make the surface usable, and a variable payload cost for the data itself.

The MCP's ramp is tiny — a single tool schema (~2.75k characters) loads on demand. But its responses land directly in context: every capped, safety-wrapped payload is tokens the model must read. The REST API inverts both. Its ramp is heavy — the agent must read ~24k characters of endpoint and quirk documentation before the first call — but a disciplined agent writes raw responses to disk and pulls only the extracted values into context. Bulk data costs almost nothing.

So the arithmetic is simple. When the payload is small, the API's documentation tax dominates and the MCP wins. When the payload is large, the API's file-first pattern keeps hundreds of kilobytes off-context while the MCP is forced to stream all of it in — and the API wins, often by a wide margin.

Rule of thumb

Reach for the MCP for narrow, interactive, read-only questions. Reach for the REST API for bulk pulls, full-history exports, raw-text analysis, or anything that writes. The break-even is roughly the point where a task's data payload would exceed the API's ~24k-character documentation ramp — a threshold a single multi-section report clears easily.

What happened. The MCP answered in a single call. The API answered too — but ~24k of its ~25k context characters were the documentation it had to read first. For five weekly data points, that ramp never pays for itself: the MCP came in ~29% cheaper.

Where they differed on capability. The MCP has no custom date range — only fixed presets — so its “90 days” was really the 3m preset. Two findings that outranked the token score

Identity: the first API attempt failed outright — its key resolved to a different workspace that couldn't see the brand. The MCP could. Same brand name, different door.
Data reality: the requested 90 days is unfulfillable by either tool — tracking only began in late May, leaving a genuine ~41-day front gap. Neither tool can invent history that was never recorded.

Three traps both surfaces share

No rank field. Rank is always a client-side computation from visibility. The MCP's own_brand_rank returns 1 for every topic — it pins the own brand to the top of the leaderboard for display. Trust it blindly and you would report De'Longhi as #1 in all seven topics; it is actually #1 in only four.
Aggregate rank ≠ daily rank. A topic can be aggregate rank 1 yet daily rank 2 on most days, because the aggregate is period-weighted share-of-voice, not an average of daily positions.
Truncation & pool size. The MCP caps competitor leaderboards at the top 25; the true pools run 44–84 brands. This is what caused the only real disagreement — a one-position wobble in the low-visibility long tail.

Practical guidance

Which door to open

Choose the MCP when

  • You are asking a narrow, read-only question interactively
  • You want shopping, per-term sources, or a saved dashboard view
  • You value fewer round-trips and no shell scaffolding
  • The answer is a handful of numbers, not a full export

Choose the REST API when

  • You are pulling many topics/competitors or full history
  • You need an exact custom date window or raw answer text
  • You need to write — manage terms, run prompts, watch credits
  • You will re-analyse the same large dataset repeatedly (write once, extract many)

They are complements, not rivals. A mature workflow uses the MCP for exploration and quick reads, and the API for scheduled exports, bulk analysis, and any change to the workspace.

Methodology & caveats

How this was measured — and its limits

Each task was run by a pair of AI sub-agents with identical model and settings, differing only in the surface they were permitted to use. Token counts, tool calls, and context volume are the harness's own accounting; context-character figures are the agents' self-reported ledgers and are approximate.

Honest limits worth stating plainly: this is one brand with sporadic, sub-three-month tracking — not a broad benchmark. The absolute token numbers reflect this agent harness and would shift under a different one; the direction of the crossover is the durable finding, not the exact figures. The REST API's token advantage on bulk pulls depends on an agent actually following the file-first discipline — a naive integration that reads full payloads into context would forfeit it. And both surfaces were exercised as implemented today; capability gaps (for example, the API skill not exposing shopping data) reflect current coverage, not permanent limits.
