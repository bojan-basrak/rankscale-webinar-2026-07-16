Part 2 of 2

Advanced analysis - past the web UI

The dashboard shows one cut at a time. The API lets you re-run the same report with filters and combine the cuts into tables that don't exist as screens. The custom metric used throughout is Brand Rank (by Visibility): the brand's position among all detected brands for a given cut, sorted by Visibility, rank 1 = most visible. It is computed from the competitor list on every call - deliberately not read from any ready-made rank field (see the method notes at the bottom for why).

Method & caveats

How these charts were made

83 API calls in total: 1 topics list, 1 search-terms report (for the prompts), 1 overall daily report, 1 citations pull, 7 per-topic reports, 9 per-engine reports, and 63 per-cell reports for the pivot. Every response was saved to a local JSON file first and all tables were computed from the files - the page never re-queries the API. The whole pull runs in about a minute, well inside the 200 requests/minute limit.

The only request shape that matters, with filters swapped per cut:

POST /v1/metrics/report
{
  "brandId": "<id>",
  "isoStartDate": "2026-05-28",
  "isoEndDate": "2026-07-14",
  "aggregation": "daily",
  "includeNotFoundExecutions": true,
  "selectedTopic": "<topic id>",     // per-topic and pivot cuts
  "selectedEngine": "<engine id>"    // per-engine and pivot cuts
}
Read with these in mind
  • The end date is exclusive. Asking the API for a window ending 07-14 returns data through 07-15. We keep the requested end date and slice the extra day off client-side rather than compensating in the request.
  • Aggregate ≠ mean of dailies. The window Visibility of 56.7 is Rankscale's period-weighted figure; the simple mean of the 29 daily values is 50.0. Headlines use the aggregate, trend reading uses the dailies.
  • Brand Rank is computed, not read. Rank = 1 + (brands with strictly higher Visibility in the cut). The API's ready-made rank fields don't reproduce the dashboard reliably (the MCP's own_brand_rank is always 1), so we don't use them.
  • Competitor daily series are short. The per-competitor time series only reaches back ~2 weeks regardless of the requested window - verified by re-querying the first block alone. Window aggregates are complete; only the competitor lines in chart B5 are truncated.
  • The 20-day gap stays a gap. No interpolation anywhere on this page.
  • One field oddity: the daily brandNotFound flag is true on all 29 days even though detection rate is 62% - it doesn't match the observable data and we ignored it.

Data: De'Longhi (US), first tracked day 2026-05-28 through 2026-07-14, pulled 2026-07-15 via the rankscale-api-skill. Raw responses cached as local JSON; reproducible against the same workspace.
